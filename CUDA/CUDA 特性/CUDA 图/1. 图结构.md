
一个操作在图中形成一个节点。操作之间的依赖关系是边。这些依赖关系限制了操作的执行顺序。

一旦操作所依赖的节点完成，该操作可在任何时间被调度。调度由CUDA系统负责。

## 1. 节点类型

 图节点可以是以下之一：

 + 内核

 + CPU函数调用

 + 内存复制

 + 初始化

 + 空节点

 + 等待一个 CUDA 事件

 + 记录一个 CUDA 事件

 + 发出外部信号量

 + 等待外部信号量

 + 条件节点

 + 内存节点

 + 子图：执行一个单独的嵌套图


## 2. 边缘数据

CUDA 12.3在CUDA图中引入了边缘数据。目前，非默认边缘数据的唯一用途是启用可编程依赖启动。

```
1. 先明确：CUDA 图是什么？

CUDA 图是把 GPU 任务（核函数、内存拷贝等）的依赖关系抽象成 “节点 + 边” 的静态流程图 ——节点 = 具体任务，边 = 任务间的依赖关系
（比如 “节点 A 执行完，节点 B 才能开始”）。

早期 CUDA 图的 “边” 是 “无数据的纯依赖”：只表示 “先后顺序”，边本身不携带任何信息，依赖逻辑是固定的（要么等前序节点全完成，
要么满足简单条件）。

2. 核心变化：边缘数据（Edge Data）是什么？

CUDA 12.3 给 CUDA 图的 “边” 增加了 “携带数据” 的能力 —— 边缘数据就是附着在 “依赖边” 上的自定义信息（比如整数标记、状态参数等）。
默认情况下，边依然是 “无数据” 的（保持兼容性）；而非默认边缘数据，目前唯一的用途就是支撑 “可编程依赖启动”。

3. 关键用途：可编程依赖启动（Programmable Dependency Launch）

早期依赖启动是 “固定规则”（比如 “前序节点全部完成才触发下一个”），而 “可编程” 意味着：
可以通过自定义逻辑，基于边缘数据判断是否启动后续节点。
比如：
(1) 给边缘数据设置 “阈值”，只有前序节点的执行结果（通过边缘数据传递）满足阈值时，才启动下一个节点；

(2)基于边缘数据的标记，动态选择启动多个候选节点中的某一个；

(3)突破 “全量依赖” 限制，实现 “部分满足条件即启动” 的灵活逻辑。

本质是让 CUDA 图的依赖触发从 “固定规则” 变成 “可自定义逻辑”，边缘数据就是传递 “判断依据” 的载体。
```
